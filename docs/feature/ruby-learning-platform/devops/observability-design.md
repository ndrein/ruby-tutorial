# Observability Design — Ruby Learning Platform

**Feature ID**: ruby-learning-platform
**Phase**: DEVOPS
**Date**: 2026-03-10
**Status**: Accepted

---

## Overview

Observability is implemented entirely with free tools: structured JSON logs emitted to stdout, captured by Fly.io's built-in log aggregation, and a health check endpoint. No paid third-party observability services are used.

---

## Structured JSON Logging

### Approach: Lograge gem

Add the `lograge` gem to the Gemfile to replace Rails' default multi-line request log format with single-line JSON.

```ruby
# Gemfile
gem "lograge"
```

### Configuration

```ruby
# config/environments/production.rb

config.log_level = :info
config.log_formatter = ::Logger::Formatter.new

config.lograge.enabled = true
config.lograge.formatter = Lograge::Formatters::Json.new
config.lograge.logger = ActiveSupport::Logger.new($stdout)

config.lograge.custom_options = lambda do |event|
  {
    request_id: event.payload[:headers]&.fetch("action_dispatch.request_id", nil),
    user_id:    event.payload[:user_id],
    host:       event.payload[:host],
    ip:         event.payload[:ip]
  }.compact
end

# Ensure all logs go to stdout for Fly.io capture
config.logger = ActiveSupport::Logger.new($stdout)
config.logger.formatter = config.log_formatter

# Force SSL (Fly.io terminates TLS, but enforce redirect)
config.force_ssl = true
```

### Log Format Specification

Every request log line is a single JSON object:

```json
{
  "method": "POST",
  "path": "/exercises/1/submit",
  "format": "html",
  "controller": "ExercisesController",
  "action": "submit",
  "status": 200,
  "duration": 47.3,
  "db": 12.1,
  "view": 8.4,
  "request_id": "a3f8b2c1-4d5e-6789-abcd-ef0123456789",
  "user_id": "uuid-here",
  "host": "ruby-learn-prod.fly.dev",
  "ip": "2a01:x:y:z",
  "time": "2026-03-10T02:15:33.421Z"
}
```

| Field | Source | Notes |
|-------|--------|-------|
| `method` | Rack | HTTP verb |
| `path` | Rack | Request path (no query string by default) |
| `format` | Rails | Response format |
| `controller` | Rails | Controller class name |
| `action` | Rails | Controller action name |
| `status` | Rails | HTTP response status code |
| `duration` | Lograge | Total request time in milliseconds |
| `db` | Lograge | Database query time in milliseconds |
| `view` | Lograge | View rendering time in milliseconds |
| `request_id` | Action Dispatch | UUID assigned per request; correlates log lines |
| `user_id` | Custom (from session) | Current user UUID; nil for unauthenticated requests |
| `host` | Rack | Request Host header |
| `ip` | Rack | Client IP (IPv4 or IPv6) |
| `time` | Lograge | ISO 8601 timestamp |

### Exposing `user_id` in Lograge

In `ApplicationController`:

```ruby
# app/controllers/application_controller.rb
def append_info_to_payload(payload)
  super
  payload[:user_id] = current_user&.id
end
```

### Background Job Logging

Solid Queue logs job start/completion through Rails logger. Add structured context for job runs:

```ruby
# app/jobs/application_job.rb
class ApplicationJob < ActiveJob::Base
  around_perform do |job, block|
    Rails.logger.info({
      event: "job_start",
      job_class: job.class.name,
      job_id: job.job_id,
      time: Time.current.iso8601
    }.to_json)

    start = Process.clock_gettime(Process::CLOCK_MONOTONIC)
    block.call
    duration_ms = ((Process.clock_gettime(Process::CLOCK_MONOTONIC) - start) * 1000).round(1)

    Rails.logger.info({
      event: "job_complete",
      job_class: job.class.name,
      job_id: job.job_id,
      duration_ms: duration_ms,
      time: Time.current.iso8601
    }.to_json)
  end
end
```

This produces log lines like:

```json
{"event":"job_start","job_class":"QueueBuilderJob","job_id":"abc123","time":"2026-03-10T02:00:01.123Z"}
{"event":"job_complete","job_class":"QueueBuilderJob","job_id":"abc123","duration_ms":342.7,"time":"2026-03-10T02:00:01.466Z"}
```

## Health Check Endpoint

### Endpoint

```
GET /up
```

Rails 7.2+ includes `Rails::HealthController` mapped to `/up` by default in `config/routes.rb`:

```ruby
# config/routes.rb
# This line is generated by Rails and should be present:
get "/up" => "rails/health#show", as: :rails_health_check
```

The controller returns:
- `200 OK` with body `{"status":"ok"}` when the application is running and the database is reachable.
- `500 Internal Server Error` if the database is unreachable.

No custom health controller is needed. The Rails default checks ActiveRecord connectivity, which is the critical dependency.

### Fly.io Health Check Integration

The `fly.toml` health check configuration:

```toml
[[http_service.checks]]
  interval = "30s"      # Check every 30 seconds
  timeout = "10s"       # Fail if no response within 10 seconds
  grace_period = "30s"  # Allow 30 seconds after startup before checking
  method = "GET"
  path = "/up"
```

Fly.io marks the machine healthy when `/up` returns 2xx. During the Recreate deployment, Fly waits for the health check to pass before marking the deploy complete. If the health check fails after 5 attempts, Fly marks the deploy failed and the release command output is available in deploy logs.

## Fly.io Built-in Log Aggregation

### Accessing Logs

```bash
# Live log stream (tails stdout from all machines)
fly logs --app <APP_NAME>

# With filtering (grep for errors)
fly logs --app <APP_NAME> | grep '"status":5'

# Historical logs (last N lines)
fly logs --app <APP_NAME> -n 100
```

Fly.io captures everything written to stdout. Because `RAILS_LOG_TO_STDOUT=true` and Lograge writes JSON to stdout, all request logs are automatically captured.

### Log Retention

Fly.io free tier retains logs for a rolling window (typically the last few thousand lines or hours — check current Fly.io documentation for exact retention). For longer retention, logs can be forwarded to a free log sink:

- Vector agent (open source) → write to local file (not available on Fly Machines without persistent volume)
- For a single-user personal tool, Fly's built-in log tailing is sufficient

### No Paid Log Aggregation

Datadog, Papertrail, Logtail, and similar services are excluded per free-tier constraint. Fly's built-in `fly logs` command is the sole log access mechanism.

## Rails Request Timing

The `duration` field in each Lograge log line provides per-request timing. No additional instrumentation is needed to track the AC-005-02 SLI (exercise submission response time < 500 ms).

To spot-check performance:

```bash
fly logs --app <APP_NAME> | grep '"action":"submit"' | python3 -c "
import sys, json
for line in sys.stdin:
    try:
        data = json.loads(line.strip())
        if 'duration' in data:
            print(f'{data[\"duration\"]} ms - {data[\"path\"]}')
    except: pass
"
```

## Solid Queue Job Timing

The `job_complete` log events (see Background Job Logging above) include `duration_ms`. Monitor nightly job completion:

```bash
# Check if nightly job ran and completed
fly logs --app <APP_NAME> | grep '"job_class":"QueueBuilderJob"'
```

Expected output for successful nightly run:
```
{"event":"job_start","job_class":"QueueBuilderJob",...,"time":"2026-03-10T02:00:01Z"}
```

## What Is Explicitly Excluded

The following are outside scope for this deployment — they would require paid services or infrastructure that violates the free-tier constraint:

- Prometheus / Grafana metrics stack
- Datadog APM
- New Relic
- Honeybadger (paid tier)
- Sentry (beyond free tier event limit)
- Papertrail / Logtail log aggregation
- Custom metrics pipeline (StatsD, OpenTelemetry collector)
- Distributed tracing (Jaeger, Zipkin)

The observability posture appropriate for a single-user personal tool is: structured logs readable on-demand + health check + Fly.io dashboard metrics. This is sufficient.
